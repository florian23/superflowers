# ADR-004: Use PostgreSQL for persistence

## Status
Accepted

Supersedes [ADR-002](ADR-002-updated.md)

## Context
The system has evolved from a single-user internal tool to a multi-service architecture. Multiple services now need to perform concurrent writes to the shared database. SQLite (chosen in ADR-002) does not support concurrent writes — it uses file-level locking, meaning only one writer can operate at a time. This has become a bottleneck causing write contention and failed transactions under the current workload.

## Decision
Replace SQLite with PostgreSQL as the persistence layer.

PostgreSQL provides robust support for concurrent writes through its MVCC (Multi-Version Concurrency Control) implementation, row-level locking, and mature connection pooling. It is well-suited for multi-service environments where multiple processes write simultaneously.

## Consequences

### Easier
- Concurrent writes from multiple services work reliably without contention
- Row-level locking allows high write throughput
- Path to horizontal scaling (read replicas, partitioning) is available if needed
- Rich ecosystem of monitoring, backup, and management tooling

### Harder
- Requires running and maintaining a database server (or managed service)
- More complex deployment and configuration compared to a single file
- Backups require `pg_dump` or continuous archiving instead of simple file copy
- Development setup now requires a running PostgreSQL instance (mitigated with Docker or testcontainers)
- Connection management (pooling, timeouts) must be configured properly
