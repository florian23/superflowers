# ADR-001: REST als API-Stil fuer die Booking API

## Status

Accepted

## Kontext

Fuer die Booking API muss ein API-Stil gewaehlt werden. Die Booking API stellt CRUD-Operationen fuer Buchungen bereit und wird primaer von Browser-Clients konsumiert.

Drei Optionen wurden im Team betrachtet:

- **REST** (Representational State Transfer)
- **GraphQL**
- **gRPC**

## Entscheidung

Wir verwenden REST fuer die Booking API.

## Begruendung

- **Team-Erfahrung:** Das Team hat umfangreiche Erfahrung mit REST-APIs. GraphQL und gRPC wuerden eine Einarbeitungsphase erfordern.
- **Einfache CRUD-Operationen:** Die Booking API bildet primaer CRUD-Operationen ab. REST ist dafuer ein natuerlicher Fit. GraphQL wuerde hier unnoetige Komplexitaet einfuehren, da keine komplexen oder verschachtelten Abfragen noetig sind.
- **Browser-Clients:** Die API wird von Browser-Clients konsumiert. REST ist im Browser-Umfeld der Standard. gRPC hat eingeschraenkten Browser-Support und erfordert einen Proxy (z.B. gRPC-Web).

## Verworfene Alternativen

### GraphQL

- Vorteile: Flexible Abfragen, kein Over-/Underfetching
- Verworfen weil: Fuer einfache CRUD-Operationen ueberdimensioniert. Hoehere Komplexitaet bei Schema-Definition und Resolver-Implementierung. Keine Team-Erfahrung vorhanden.

### gRPC

- Vorteile: Hohe Performance, starke Typisierung via Protobuf, effizientes Streaming
- Verworfen weil: Eingeschraenkter Browser-Support. Erfordert gRPC-Web Proxy fuer Browser-Clients. Keine Team-Erfahrung vorhanden. Performance-Vorteile bei einfachen CRUD-Operationen nicht ausschlaggebend.

## Konsequenzen

- API folgt REST-Konventionen (HTTP-Methoden, Statuscodes, Ressourcen-URIs).
- OpenAPI/Swagger wird zur Dokumentation verwendet.
- Sollten sich die Anforderungen aendern (z.B. komplexe verschachtelte Abfragen, Service-to-Service-Kommunikation mit hohen Performance-Anforderungen), muss diese Entscheidung neu bewertet werden.
