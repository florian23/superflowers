# ADR-001: Use Event-Driven architecture for IoT platform

## Status
Accepted

## Context
We are building an IoT platform deployed on AWS with a team of 4 developers. The platform must handle high-volume sensor data ingestion, real-time event processing, and asynchronous device-to-cloud communication. We evaluated three architecture styles using a weighted scoring matrix against our driving characteristics:

**Alternatives considered:**

| Style | Score (weighted) | Estimated AWS cost (monthly, 10k devices) |
|-------|-----------------|-------------------------------------------|
| Event-Driven | 15/15 | ~$2,000–$4,000 ($$$ tier) |
| Microservices | 15/15 | ~$5,000–$10,000 ($$$$$ tier) |
| Space-Based | 13/15 | ~$4,000–$7,000 ($$$$ tier) |

**Event-Driven** and **Microservices** tied on fitness scores (15/15), but diverge sharply on cost and team capacity:

- **Microservices** requires per-service CI/CD pipelines, independent deployability infrastructure, service mesh or API gateway, and distributed tracing. With 4 developers, each person would own 3–5 services — an unsustainable operational burden. AWS costs are 2–3x higher due to per-service compute, load balancers, and observability tooling.
- **Space-Based** scored 13/15 — lower on simplicity and cost. Requires in-memory data grids (e.g., Hazelcast/Redis clustering) with complex tuning. The 2-point gap reflects weaker alignment with our cost and maintainability requirements.
- **Event-Driven** leverages managed AWS services (EventBridge, SQS, Lambda) that reduce operational overhead. A 4-person team can manage event producers and consumers without maintaining service mesh infrastructure.

## Decision
We will use Event-Driven architecture because it matches the top fitness score (15/15) at the lowest operational cost, and its managed-service model on AWS is sustainable for a 4-developer team.

## Consequences
**What gets easier:**
- Sensor data ingestion scales naturally via event streams (SQS/Kinesis) without provisioning per-device infrastructure
- Adding new event consumers (analytics, alerting, storage) requires no changes to existing producers — loose coupling by design
- AWS managed services (EventBridge, Lambda, SQS) reduce operational burden; the team spends time on business logic, not infrastructure
- Monthly AWS costs stay in the $2k–$4k range vs $5k–$10k for microservices — saving $36k–$72k/year

**What gets harder:**
- Debugging distributed event flows is harder than tracing synchronous request/response — we will need structured correlation IDs and centralized logging (CloudWatch/X-Ray) from day one
- Eventual consistency is inherent — if any workflow requires strong consistency (e.g., device provisioning confirmation), we must design explicit choreography or use Step Functions, adding complexity
- Event schema evolution requires discipline; breaking changes to event payloads can silently break downstream consumers unless we enforce schema validation (e.g., EventBridge schema registry)
- Testing event-driven flows end-to-end is more complex than testing REST APIs — requires event replay tooling and consumer contract tests
- If we later need request/response patterns (e.g., synchronous device commands), we will need to layer in an API Gateway, partially negating the pure event-driven simplicity
