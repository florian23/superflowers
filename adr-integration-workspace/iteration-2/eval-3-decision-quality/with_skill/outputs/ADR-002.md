# ADR-002: Use PostgreSQL for financial services data persistence

## Status
Accepted

## Context
We are building a financial services application with strict regulatory requirements (SOX/PCI-DSS audit trails, data lineage). The system processes financial transactions that demand ACID guarantees — partial writes or phantom reads are not acceptable in account balance calculations or ledger entries. Our team has 3+ years of SQL expertise and existing tooling (migrations, ORMs, query analyzers).

**Alternatives considered:**

**1. PostgreSQL (relational, ACID-compliant)**
- Pros: Full ACID transactions with serializable isolation; native audit capabilities (pgAudit extension); row-level security for multi-tenant compliance; mature ecosystem for financial patterns (e.g., advisory locks, CTEs for ledger queries); team has deep SQL expertise — no ramp-up cost
- Cons: Vertical scaling has limits (estimated ceiling ~50k TPS on r6g.4xlarge before read replicas needed); schema migrations require planning for zero-downtime deploys; operational burden of managing replication, backups, and failover (mitigated by AWS RDS)

**2. MongoDB (document store)**
- Pros: Flexible schema for varied financial product types; horizontal scaling via sharding; multi-document transactions available since v4.0
- Cons: Multi-document transactions have higher latency (~2–5x vs PostgreSQL single-node) and are limited to 16MB per transaction; no native row-level security — audit trail implementation is application-level; eventual consistency is the default — must explicitly configure read/write concerns for strong consistency; team would need 2–3 months ramp-up on document modeling for financial data; regulatory auditors are less familiar with MongoDB audit patterns, increasing audit friction

**3. DynamoDB (managed NoSQL)**
- Pros: Fully managed; near-unlimited horizontal scale; single-digit millisecond reads; pay-per-request pricing at low volumes
- Cons: Transactions limited to 25 items per transaction group — insufficient for multi-account settlement batches that routinely involve 50–200 items; no ad-hoc SQL queries — regulatory auditors require ability to run arbitrary queries against transaction history; single-table design pattern has steep learning curve; no foreign key constraints — referential integrity is entirely application-managed, increasing risk of orphaned records in ledger data

## Decision
We will use PostgreSQL because it provides native ACID transactions at the isolation levels our financial workflows require, our team's existing SQL expertise eliminates ramp-up cost, and its audit ecosystem (pgAudit, row-level security) directly satisfies regulatory audit requirements without application-level workarounds.

## Consequences
**What gets easier:**
- Transaction integrity is guaranteed at the database level — no application-level compensation logic needed for multi-account operations
- Regulatory audits are streamlined: pgAudit provides statement-level logging, auditors can run ad-hoc SQL queries against any historical data, and row-level security enforces tenant isolation at the database layer
- Team productivity is immediate — no ramp-up time on a new data modeling paradigm; existing migration tooling (Flyway/Liquibase) and query profiling (pg_stat_statements) apply directly
- Financial reporting queries (aggregations, window functions, CTEs) are first-class citizens in SQL

**What gets harder:**
- Horizontal scaling beyond ~50k TPS requires read replicas or partitioning strategies (table partitioning by date for ledger data); this is adequate for our projected 5-year volume but creates a scaling ceiling that MongoDB/DynamoDB would not have
- Schema changes on large tables (100M+ rows) require careful migration planning — online DDL tools (pg_repack) or blue-green deploy strategies to avoid locking production tables
- We accept vendor coupling to the PostgreSQL ecosystem — switching databases later would require rewriting queries, audit infrastructure, and row-level security policies; estimated migration cost: 3–6 months of engineering effort
- Geographic distribution (multi-region active-active) is harder than with DynamoDB; we would need Citus or Aurora Global Database, adding complexity and cost
