# ADR-003: Accept eventual consistency for social media feed reads

## Status
Accepted

## Context
Our social media platform's feed service is the highest-traffic read path in the system. Users consume feed content at a rate that demands high read throughput, while tolerating minor staleness — a post appearing 500ms late in a friend's feed is imperceptible to users, but the throughput difference between consistency models is dramatic.

**Measured performance data:**

| Consistency Model | Read throughput | P99 read latency | Cache hit rate |
|-------------------|----------------|-------------------|----------------|
| Strong consistency | ~5,000 reads/sec | 45ms | N/A (no caching) |
| Eventual consistency (500ms staleness window) | ~50,000 reads/sec | 8ms | ~95% |

The 10x throughput gain comes from aggressive read-through caching with a 500ms TTL. Strong consistency prohibits caching (every read must hit the primary datastore to guarantee freshness), creating a hard throughput ceiling.

**Alternatives considered:**

**1. Strong consistency (linearizable reads)**
- Pros: Every read reflects the latest write — no stale data; simpler mental model for developers; eliminates an entire class of bugs (stale read race conditions)
- Cons: Throughput ceiling at ~5,000 reads/sec requires proportional scaling of primary datastore infrastructure; P99 latency 5–6x higher (45ms vs 8ms); no caching layer possible — every read is a database hit; estimated infrastructure cost 8–10x higher to match eventual consistency throughput via horizontal scaling of primary nodes

**2. Eventual consistency with 500ms staleness window (chosen)**
- Pros: 10x read throughput (50,000 reads/sec) via read-through caching; P99 latency drops from 45ms to 8ms; infrastructure cost reduced by ~80% at equivalent throughput; 95% cache hit rate means 95% of reads never touch the database
- Cons: Posts, likes, and comments may appear up to 500ms late in other users' feeds; users editing their own post may briefly see the old version (read-your-own-writes violation); cache invalidation bugs can extend staleness beyond 500ms if TTL or invalidation logic fails

**3. Session consistency (read-your-own-writes guarantee)**
- Pros: Users always see their own updates immediately; still allows caching for other users' reads; better UX for content creators
- Cons: Requires per-user session affinity or versioned reads — adds routing complexity; cache efficiency drops to ~70–80% (user's own content bypasses cache); throughput gain reduced to ~6–7x vs 10x; adds complexity to load balancing (sticky sessions or version vectors)

## Decision
We will accept eventual consistency with a 500ms staleness window for feed reads because the 10x throughput gain (5,000 to 50,000 reads/sec) and 80% infrastructure cost reduction outweigh the imperceptible delay in content freshness for a social media feed.

## Consequences
**What gets easier:**
- Read infrastructure scales to 50,000 reads/sec with minimal database load — 95% of reads served from cache at 8ms P99 latency
- Infrastructure costs drop ~80% compared to strong consistency at equivalent throughput — the caching layer (Redis/Memcached) is far cheaper than scaling primary database nodes
- Adding new feed features (new post types, reactions) only requires cache key changes, not database scaling
- Feed rendering latency improves from 45ms to 8ms at P99, directly improving user-perceived performance

**What gets harder:**
- Users may see stale data for up to 500ms — a user posts a comment and their friend does not see it for half a second; this is acceptable for feed browsing but would be unacceptable for direct messages or notifications (those paths must use strong consistency)
- Read-your-own-writes is not guaranteed — after posting, the author may briefly see their feed without the new post; we must implement client-side optimistic updates to mask this (injecting the post into the local feed state before the server confirms)
- Cache invalidation failures can extend staleness beyond 500ms — we need monitoring and alerting on cache TTL violations (p99 staleness > 1s should trigger an alert)
- Any future feature requiring strong consistency on the feed path (e.g., real-time auction bidding in feed) cannot use this caching layer — it will need a separate read path, increasing architectural complexity
- Debugging consistency-related bugs is harder — reproducing "I posted but my friend didn't see it" requires correlating timestamps across cache layers, which demands structured logging with sub-second precision
