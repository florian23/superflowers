# ADR-002: PostgreSQL for Financial Services Data Store

## Status

Accepted

## Date

2026-03-29

## Context

We are selecting a primary database for a financial services application that handles transactions, account balances, and regulatory reporting. The system must support ACID transactions, pass regulatory audits, and be maintainable by a team with strong SQL expertise. We evaluated PostgreSQL, MongoDB, and DynamoDB.

### Key Requirements

1. **ACID compliance**: Financial transactions require atomicity and consistency guarantees. Partial writes or inconsistent reads on monetary data are unacceptable.
2. **SQL expertise**: The team has deep SQL knowledge. Adopting a non-SQL database would reduce productivity and increase onboarding time.
3. **Regulatory audits**: Financial regulators require demonstrable data integrity, audit trails, and the ability to run ad-hoc queries against historical data for compliance reporting.

## Decision

We will use **PostgreSQL** as the primary data store.

## Rationale

### ACID Transactions

PostgreSQL provides full ACID transaction support with serializable isolation when needed. Financial operations such as transfers between accounts require multi-row atomic updates -- a fundamental capability of PostgreSQL. MongoDB added multi-document transactions in v4.0 but they carry performance penalties and are not its primary design strength. DynamoDB supports transactions across up to 100 items but with significant throughput cost (2x capacity units) and a 4MB size limit per transaction.

### SQL Expertise

The team has years of SQL experience. PostgreSQL allows the team to be immediately productive with complex queries, joins, window functions, and CTEs that financial reporting demands. MongoDB would require learning an aggregation pipeline syntax. DynamoDB's query model is fundamentally different -- single-table design patterns and GSIs require a paradigm shift that would slow development.

### Regulatory Audit Support

Financial regulators expect:
- **Schema enforcement**: PostgreSQL's strict schema ensures data integrity at the database level. Auditors can inspect table definitions to verify data constraints.
- **Ad-hoc querying**: Compliance officers and auditors need to run arbitrary queries. SQL is the industry standard for this. MongoDB's query language is less familiar to auditors. DynamoDB's query capabilities are limited to primary key and index access patterns.
- **Audit trails**: PostgreSQL supports triggers, logical replication, and extensions like `pgaudit` for comprehensive audit logging that meets regulatory standards.

## Alternatives Considered

### MongoDB (Rejected)

MongoDB's document model offers flexibility but works against us in a financial context. Schema-less storage creates risk of inconsistent data shapes that complicate auditing. While MongoDB supports transactions, they are a bolted-on feature rather than a core design principle. The aggregation pipeline, while powerful, is unfamiliar to our SQL-skilled team and to the auditors who review our systems.

### DynamoDB (Rejected)

DynamoDB excels at high-throughput key-value access patterns but is poorly suited for financial services. Its limited transaction support, lack of ad-hoc SQL querying, and single-table design patterns make regulatory compliance harder to demonstrate. The eventual consistency default mode is inappropriate for financial data without careful per-query configuration. Cost can also become unpredictable with complex access patterns requiring multiple GSIs.

## Consequences

### Positive

- Full ACID guarantees protect financial data integrity
- Team is immediately productive with existing SQL skills
- Rich ecosystem of tools for auditing, monitoring, and reporting
- Strong community and long-term stability (PostgreSQL has 25+ years of development)
- Extensions like PostGIS, pg_partman, and pgaudit provide capabilities without leaving the PostgreSQL ecosystem

### Negative

- Vertical scaling has limits compared to DynamoDB's horizontal scaling (mitigated by read replicas and partitioning)
- Operational burden of managing PostgreSQL instances (mitigated by using a managed service such as AWS RDS or Aurora PostgreSQL)
- Schema migrations require careful planning in production (mitigated by tools like Flyway or Liquibase)

### Risks

- If read throughput requirements grow significantly, PostgreSQL may need read replicas or caching layers. Mitigation: design the data access layer to support read replicas from the start.
