# ADR-003: Eventual Consistency for Social Media Feed

## Status

Accepted

## Date

2026-03-29

## Context

We are designing the consistency model for a social media feed system. The feed aggregates posts from followed users and must serve read-heavy traffic at scale. We evaluated two approaches: eventual consistency with up to 500ms stale reads, and strong consistency with always-current reads.

### Traffic Profile

The feed is extremely read-heavy. Users refresh and scroll feeds far more often than they create posts. Optimizing for read throughput directly impacts user experience and infrastructure cost.

### The Tradeoff

| Property | Eventual Consistency | Strong Consistency |
|---|---|---|
| Read Throughput | 10x baseline | 1x baseline |
| Data Staleness | Up to 500ms | None |
| Read Latency (p99) | Low (cached replicas) | Higher (coordination required) |
| Infrastructure Cost | Lower per read | Higher per read |

## Decision

We will adopt **eventual consistency** for the social media feed, accepting up to 500ms of stale reads in exchange for 10x read throughput.

## Rationale

### 500ms Staleness Is Imperceptible for Feeds

A social media feed is not a financial ledger. If a user sees a friend's post 500ms after it was published rather than immediately, the experience is indistinguishable. Users do not have a real-time expectation for feed content -- they already accept that feeds are algorithmically ordered and not strictly chronological.

### 10x Read Throughput Changes the Economics

Strong consistency requires coordination on every read (checking that the data is current, potentially waiting for replication). Eventual consistency allows serving reads from local replicas and caches without coordination. The 10x throughput improvement means:

- **Fewer servers**: Serving the same user base requires roughly 1/10th the read infrastructure.
- **Lower latency**: Reads served from local replicas avoid cross-region or cross-node round trips.
- **Better degradation**: Under load spikes, eventual consistency degrades gracefully (slightly staler data) rather than failing or queuing requests.

### Strong Consistency Costs Are Unjustifiable for This Use Case

Enforcing strong consistency on a feed system means every read must verify it has the latest data. At social media scale, this coordination overhead is enormous and provides no meaningful user benefit. The cost-to-value ratio is deeply unfavorable.

## Alternatives Considered

### Strong Consistency (Rejected)

Strong consistency guarantees that every read returns the most recent write. For a social media feed, this guarantee provides negligible user value at 10x the infrastructure cost. Strong consistency is the correct choice for systems where stale reads cause correctness problems (financial transactions, inventory counts, authentication). A social media feed does not have this property.

## Consequences

### Positive

- 10x read throughput enables serving more users with less infrastructure
- Lower read latency improves perceived feed performance
- Simpler read path (no coordination) reduces failure modes
- Cost savings scale linearly with user growth

### Negative

- Users may occasionally see a feed that is up to 500ms behind the latest state
- New posts from followed users may appear with a brief delay
- Write-then-read patterns (post something, immediately check the feed) may not reflect the new post instantly (mitigated by read-your-own-writes consistency for the authoring user)

### Risks

- If product requirements evolve to need real-time feed updates (e.g., live event feeds), 500ms staleness may become noticeable. Mitigation: implement a WebSocket push channel for real-time scenarios, keeping eventual consistency as the default for standard feed reads.
- Debugging issues becomes harder when data is not guaranteed to be consistent across replicas. Mitigation: include replication lag metrics in monitoring and alerting.
