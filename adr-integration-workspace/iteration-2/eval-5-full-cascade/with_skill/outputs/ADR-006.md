# ADR-006: Use Service-Based Architecture

## Status
Accepted

## Context
This supersedes ADR-001 because the organization has grown from 1 team to 3 teams.

The modular monolith (ADR-001) served the project well as a single-team effort. However, with 3 teams now working on the system, the single deployment artifact and shared release cycle create coordination bottlenecks. Teams block each other on deployments, merge conflicts in shared modules are frequent, and independent team velocity is constrained.

The driving characteristics remain Simplicity (Critical), Testability (Critical), and Maintainability (Critical), but the organizational context has changed significantly.

Alternatives considered:
1. **Keep Modular Monolith** — Cheapest option, but team coordination overhead is already painful at 3 teams and will get worse.
2. **Microservices** — Maximum team independence, but operational complexity is high ($$$$) and the team lacks distributed systems experience. Overkill for 3 teams.
3. **Service-Based Architecture** — Coarse-grained services (one per team domain), independent deployability, moderate operational overhead. Preserves simplicity better than microservices while solving the multi-team coordination problem.

Service-based architecture scores well on the driving characteristics:
- Simplicity: 3/5 (lower than modular monolith's 5/5, but acceptable)
- Testability: 4/5 (service-level testing is straightforward)
- Maintainability: 4/5 (clear service boundaries aligned to team ownership)

The cost category moves from $ to $$, which is justified by eliminating the multi-team coordination bottleneck.

## Decision
We will use a Service-Based architecture because it enables 3 teams to develop, test, and deploy independently while maintaining reasonable simplicity. Each team owns one or more coarse-grained services aligned to business domains.

## Consequences
**Easier:**
- Teams can deploy independently without coordinating release windows
- Service boundaries align naturally with team boundaries (one service per team domain)
- Each team can choose internal patterns without affecting others
- Testability remains high — each service is testable in isolation

**Harder:**
- Inter-service communication introduces network latency and failure modes not present in the monolith
- Distributed transactions across services require saga patterns or eventual consistency
- Local development setup is more complex (multiple services to run)
- Operational overhead increases: service discovery, health checks, distributed logging

**Tradeoffs accepted:**
- Simplicity drops from 5/5 to 3/5 — accepted because the multi-team coordination pain exceeds the added architectural complexity
- Single deployment artifact is gone — accepted because independent deployability is now more valuable than deployment simplicity
- Need to define clear API contracts between services — accepted as a necessary discipline for multi-team development

**Fitness functions affected:**
- All modular monolith style FFs (referencing ADR-001) are removed and replaced with service-based style FFs referencing this ADR
- Quality scenarios involving deployment, module boundaries, and cross-module communication need re-evaluation
