# ADR-006: Use Service-Based Architecture Style

**Status:** Accepted

**Date:** 2026-03-29

**Supersedes:** [ADR-001: Use Modular Monolith Architecture Style](ADR-001-updated.md)

## Context

The organization has grown from 1 team to 3 independent teams. Under the modular monolith (ADR-001), this growth exposed several problems:

- **Deployment contention:** All 3 teams share a single deployable. A failed build or test in one module blocks releases for every team.
- **Merge conflicts:** With 3 teams committing to the same repository and build pipeline, coordination overhead is high and increasing.
- **Independent scaling:** Some domains (e.g., order processing) need to scale independently during peak periods, which a single process cannot support.
- **Blast radius:** A defect in any module can bring down the entire application.

We evaluated three candidate styles:

| Style | Pros | Cons |
|-------|------|------|
| Modular Monolith (status quo) | Simple ops, low latency | Deployment contention, no independent scaling |
| Service-Based | Independent deploy per domain, manageable ops complexity, coarse-grained services | Some network overhead, need for service contracts |
| Microservices | Maximum independence, fine-grained scaling | Excessive operational complexity for 3 teams, distributed data challenges |

Service-based architecture provides the right balance: each team owns one or more coarse-grained domain services that deploy independently, without the operational burden of fine-grained microservices.

## Decision

We adopt a **Service-Based Architecture** style with the following characteristics:

- **Coarse-grained domain services:** Each service maps to a bounded context (not a single entity). Target: 4-7 services total.
- **Independent deployability:** Each service has its own build pipeline, deployable artifact, and runtime.
- **Database-per-service:** Each service owns its data store. No shared databases.
- **Synchronous REST for queries, asynchronous messaging for commands/events** between services.
- **Shared UI layer (BFF):** A single Backend-for-Frontend aggregates service responses for the web client.
- **Service contracts via OpenAPI specs** stored in a contracts repository.

## Fitness Functions

The following fitness functions replace and extend those from ADR-001. Retired monolith-specific FFs are noted.

| ID | Fitness Function | Target | Rationale | Replaces |
|----|-----------------|--------|-----------|----------|
| FF-001 | No cyclic dependencies between services | 0 cycles | Preserve service autonomy | FF-001 (module cycles) |
| FF-002 | Service API backward compatibility | 100% backward-compatible releases | Prevent breaking consumers | FF-002 (module API surface) -- retired |
| FF-003 | Per-service build time | < 3 minutes per service | Independent, fast builds | FF-003 (full monolith build) |
| FF-004 | Service coupling (shared dependencies) | 0 shared runtime dependencies between services | Hard boundary enforcement | FF-004 (module instability) |
| FF-005 | Independent deployability | Each service deployable without coordinating with other services | Core architectural goal for multi-team | New |
| FF-006 | Inter-service latency (P99) | < 100ms per synchronous call | Network calls replace in-process; must stay bounded | New |
| FF-007 | Data ownership violation | 0 direct cross-service DB queries | Enforce database-per-service | New |

| Old ID | Name | Reason |
|--------|------|--------|

## Quality Scenario Impact

The style change affects existing quality scenarios. The table below maps each scenario to its required update.

| ID | Quality Attribute | ADR-001 Scenario | Impact | Required Change |
|----|------------------|-----------------|--------|-----------------|
| QS-001 | Deployability | Single artifact in < 5 min | **Rewrite** | Each service deployed independently in < 5 min. No cross-team deploy coordination required. |
| QS-002 | Modifiability | Change in one module, no changes elsewhere | **Refine** | Change in one service requires no changes in other services. Contract versioning absorbs interface evolution. |
| QS-003 | Testability | Full integration tests < 10 min | **Rewrite** | Per-service test suite < 5 min. Cross-service contract tests < 10 min. Full end-to-end suite < 20 min. |
| QS-004 | Performance | In-process P99 < 5ms | **Rewrite** | Intra-service P99 < 5ms. Inter-service synchronous P99 < 100ms. Async message delivery < 500ms. |
| QS-005 | Reliability | Single process failure < 0.1% | **Rewrite** | Individual service availability > 99.9%. Failure in one service does not cascade (bulkhead). |
| QS-006 | Maintainability | New dev productive in 1 week | **Refine** | New dev productive in 1 week within their team's services. Cross-service onboarding < 2 weeks. |

## Consequences

**Positive:**
- Each team deploys independently -- no deployment contention
- Failures are isolated to individual services (reduced blast radius)
- Services can scale independently based on load profile
- Teams have full ownership of their domain, including technology choices within guidelines
- Clearer domain boundaries enforced by network rather than convention

**Negative:**
- Increased operational complexity (multiple pipelines, monitoring per service)
- Network latency replaces in-process calls -- performance characteristics change
- Distributed data management: no cross-service joins, eventual consistency for some operations
- Need for contract testing and versioning discipline
- Requires investment in observability (distributed tracing, centralized logging)

## Migration Path

1. Extract bounded contexts from the monolith as services, starting with the most independent module
2. Introduce an event bus for asynchronous communication
3. Split databases per extracted service
4. Retire the monolith once all domains are extracted
5. Each extraction follows the Strangler Fig pattern to allow incremental migration
