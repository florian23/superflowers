# ADR-005: Use Microservices architecture

## Status
Accepted

## Context
This supersedes ADR-001 because the organization has grown from 2 to 5 teams and requires finer-grained independent scaling.

Under ADR-001, the system uses Service-Based architecture with 4-12 coarse-grained services. This worked well at smaller scale, but the team has encountered the following pressures:

- **5 independent teams** need to deploy independently without coordinating release cycles
- **Finer-grained scaling** is required -- some capabilities need 10x the resources of others, and the coarse service boundaries of Service-Based architecture force over-provisioning
- **Release velocity** is constrained by shared service boundaries -- changes in one domain block deployments in the same service

Alternatives considered:
1. **Stay with Service-Based** -- lower operational cost, but scaling and team independence limits are already painful
2. **Microservices** -- each service independently deployable and scalable, aligned with team boundaries, at the cost of higher operational complexity
3. **Event-Driven** -- excellent for decoupling, but addresses communication patterns rather than service granularity; can complement microservices but doesn't solve the deployment independence problem alone

## Decision
We will use Microservices architecture because the growth to 5 teams and the need for fine-grained independent scaling and deployment make the operational overhead of microservices justified. Each team will own one or more independently deployable services aligned with bounded contexts.

## Consequences
- Easier: Each team deploys independently -- no cross-team release coordination
- Easier: Fine-grained scaling -- scale only the services that need it
- Easier: Technology diversity -- teams can choose the best tool for their service
- Easier: Fault isolation -- one service failure doesn't bring down the system
- Harder: Operational complexity increases significantly (service discovery, distributed tracing, circuit breakers)
- Harder: Data consistency across services requires explicit patterns (saga, event sourcing)
- Harder: API contract management becomes critical -- breaking changes are costly
- Harder: Testing across service boundaries requires contract testing infrastructure
- Accepted tradeoff: Higher infrastructure and operational cost in exchange for team autonomy and scaling flexibility
- All fitness functions referencing ADR-001 are removed and replaced with Microservices fitness functions referencing ADR-005
