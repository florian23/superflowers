# ADR-005: Migrate to Microservices Architecture

## Status

**Accepted**

## Date

2026-03-29

## Context

Our system currently uses a service-based architecture (ADR-001) with coarse-grained services sharing a database. Two drivers necessitate a change:

1. **Team scaling**: We have grown from 2 to 5 independent delivery teams. The coarse-grained service boundaries force cross-team coordination on deployments and shared database schema changes, reducing team autonomy.
2. **Finer scaling requirements**: Traffic patterns show that specific capabilities (e.g., order processing, search) need to scale independently. The service-based model only allows scaling at the coarse service level, leading to over-provisioning.

## Decision

Migrate from the service-based architecture to a **microservices architecture** with the following characteristics:

- Each microservice owns its own datastore (database-per-service pattern)
- Services communicate via asynchronous messaging (events) for decoupled workflows, with synchronous REST/gRPC for query-response interactions
- Each of the 5 teams owns 2-4 microservices aligned to bounded contexts
- An API gateway provides a unified entry point and handles cross-cutting concerns (auth, rate limiting)
- Incremental migration using the strangler fig pattern, extracting microservices from existing coarse-grained services one bounded context at a time

## Consequences

### Positive

- Each team can deploy independently without cross-team coordination
- Individual services can scale horizontally based on their specific load
- Fault isolation: a failure in one microservice does not cascade to the entire system
- Technology flexibility: teams can choose the best datastore or language for their bounded context

### Negative

- Increased operational complexity (more deployables, distributed tracing needed)
- Data consistency shifts from ACID transactions to eventual consistency across services
- Network latency and reliability become first-class concerns
- Requires investment in platform capabilities: service mesh, observability, CI/CD per service
