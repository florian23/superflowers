# ADR-004: Notifications-Service for Mobile Push Notifications

## Status

Accepted

## Date

2026-03-29

## Context

The system needs the ability to send push notifications to mobile devices (iOS and Android). Other services in the system must be able to trigger notifications for events such as order updates, alerts, or user-relevant messages.

The existing architecture is service-based, uses PostgreSQL for persistence, and services communicate via REST APIs. A review of these decisions (see adr-review.md) confirmed that a dedicated Notifications-Service aligns with the current architecture. No existing decisions need to be revised.

Key considerations:

- Push notifications require integration with platform-specific providers: Apple Push Notification service (APNs) for iOS and Firebase Cloud Messaging (FCM) for Android.
- Notification delivery is inherently asynchronous -- the sender should not wait for device delivery.
- Device token management (registration, renewal, invalidation) must be handled reliably.
- Delivery tracking and retry logic are essential because push providers do not guarantee delivery.

## Decision

We will introduce a **Notifications-Service** as a new service in the existing service-based architecture with the following design:

### Service Responsibilities

1. **Device Registration**: Expose a REST API for mobile clients to register and update their push tokens (FCM/APNs).
2. **Notification Intake**: Accept notification requests from other services via REST API.
3. **Push Delivery**: Send push notifications to the appropriate provider (FCM for Android, APNs for iOS).
4. **Persistence**: Use PostgreSQL to store device registrations, notification logs, and delivery status.
5. **Retry & Error Handling**: Implement retry logic for transient provider failures and handle token invalidation feedback from FCM/APNs.

### Integration Pattern

- Other services call the Notifications-Service REST API to request a notification (e.g., `POST /notifications`).
- The Notifications-Service processes delivery asynchronously -- it acknowledges the request immediately and handles provider communication in the background.
- Internal asynchronous processing (e.g., via an in-process queue or worker threads) decouples the REST response from actual delivery, ensuring the calling service is not blocked.

### Data Model (PostgreSQL)

- **device_registrations**: device_id, user_id, platform (ios/android), push_token, created_at, updated_at
- **notifications**: id, recipient_user_id, title, body, payload, status (pending/sent/failed), created_at, sent_at
- **delivery_log**: id, notification_id, device_id, provider_response, status, attempted_at

### Provider Abstraction

A provider interface abstracts FCM and APNs so that:
- Adding a new push channel in the future does not require structural changes.
- Provider-specific logic (token formats, payload structures, error codes) is encapsulated.

## Consequences

### Positive

- Clean separation of concerns: notification logic is isolated in one service.
- Consistent with the service-based architecture -- no new architectural patterns required.
- PostgreSQL reuse keeps the operational footprint small.
- Provider abstraction enables future extensibility (e.g., web push, SMS).
- Asynchronous internal processing ensures calling services are not affected by provider latency or outages.

### Negative

- Adds a new service to deploy and operate.
- Introduces external dependencies on FCM and APNs with their own availability characteristics.
- Without a message queue between services, notification requests can be lost if the Notifications-Service is down when another service calls it. This is accepted for now; a message queue can be introduced later if reliability requirements increase.

### Risks

- Push token lifecycle management is error-prone (tokens expire, users reinstall apps). Robust handling of provider feedback is critical.
- Rate limits imposed by FCM/APNs must be respected to avoid throttling.
