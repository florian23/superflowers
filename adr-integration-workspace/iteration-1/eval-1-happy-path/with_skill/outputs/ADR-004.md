# ADR-004: Add Notification-Service for mobile push notifications

## Status
Accepted

## Context
The system needs to send push notifications to mobile devices (iOS and Android). This requires integrating with platform-specific push services (Apple Push Notification service, Firebase Cloud Messaging) and managing device tokens, notification templates, and delivery state.

We considered three alternatives:

1. **New standalone Notification-Service** — A dedicated service within the existing Service-Based architecture, responsible for all push notification concerns: device registration, template management, delivery via APNs/FCM, and delivery tracking.
2. **Extend an existing service** — Add notification logic to an existing service (e.g., a User-Service or Communication-Service). This avoids deploying a new service but mixes notification concerns with unrelated domain logic.
3. **Third-party notification platform** (e.g., OneSignal, Pusher) — Delegate push delivery entirely to an external SaaS. Reduces implementation effort but introduces a vendor dependency and recurring cost.

The existing architecture is Service-Based (ADR-001), uses PostgreSQL for persistence (ADR-002), and REST for inter-service communication (ADR-003). All three ADRs are compatible with adding a new service.

## Decision
We will add a dedicated Notification-Service as a new service within the Service-Based architecture. The service owns its own PostgreSQL schema for device tokens, notification logs, and delivery status. Other services trigger notifications via REST API calls to the Notification-Service. The service handles push delivery to mobile devices through APNs and FCM.

This keeps notification concerns isolated in a single service, follows the established Service-Based pattern, and avoids coupling notification logic into unrelated services.

## Consequences

**What becomes easier:**
- Notification logic is contained in one service — changes to delivery providers, templates, or retry strategies do not affect other services.
- Device token management and delivery tracking have a clear owner.
- The service can be scaled independently if notification volume grows.
- Other services have a simple integration point: POST to the Notification-Service REST API.

**What becomes harder:**
- One more service to deploy, monitor, and maintain.
- Notification delivery depends on external providers (APNs, FCM) — the service must handle provider outages, rate limits, and token invalidation gracefully.
- Debugging notification failures requires tracing across the calling service, the Notification-Service, and the external push provider.

**Constraints accepted:**
- Inter-service communication remains REST (ADR-003) — no event bus or message queue for triggering notifications. If high-throughput asynchronous delivery becomes necessary, a future ADR may introduce a message broker.
- The Notification-Service uses PostgreSQL (ADR-002) for persistence, consistent with the rest of the system.
